# Retrospective — v0.3.16

**Date:** 2026-04-05
**Milestone:** `_agent_audit` grep flag safety — dash-prefix path allowlist fix + tests
**PR:** #22 — merged to main (`c43c4f5`)
**Participants:** Claude, Copilot

## What Went Well

- Root cause was narrow and well-understood: `grep -Fqx "$file"` without `--` would interpret a path like `-infra.yaml` as a grep flag, causing an allowlist bypass
- Fix was a single-character change (`-- ` added before `"$file"`) — minimal blast radius
- Copilot caught the missing regression test before merge — the right gate at the right time
- BATS test pattern for dash-prefix paths (`git add -- -infra.yaml`, `printf '%s\n' '-infra.yaml'`) is now documented in the test suite as a reference

## What Went Wrong

- CHANGE.md was not version-bumped before merging — `[Unreleased]` was not stamped `[v0.3.16]`; fixed on next branch

## Process Rules Added

None — existing Copilot gate + BATS requirement caught the gap correctly.

## Decisions Made

- **`printf '%s\n' '<value>'` instead of `printf '<value>\n'`** — when the value starts with `-`, bare `printf` treats it as a flag option. Always use `printf '%s\n' "$value"` in BATS tests for safety.

## Theme

v0.3.16 was a one-line safety fix and two BATS tests. The `--` separator in `grep -Fqx -- "$file"` is a standard POSIX idiom for preventing argument-as-flag misparse; the fix was straightforward. The value was entirely in having a regression test — without it, the edge case would remain untested even after the fix. Copilot's comment was the forcing function. The BATS test itself hit a secondary `printf` flag-parsing issue (`-infra.yaml` as format string), which was a useful reminder to always use `printf '%s\n'` for values that may start with `-`.
