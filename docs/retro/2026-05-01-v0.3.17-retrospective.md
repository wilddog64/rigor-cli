# Retrospective — v0.3.17

**Date:** 2026-05-01
**Milestone:** `_ai_agent_review` generic dispatch + `_copilot_review` gate fix + `_agent_lint` glob expansion
**PR:** #24 — merged to main (`108924b9`)
**Participants:** Claude, Codex, Copilot

## What Went Well

- Copilot caught a real code bug (`_ai_agent_review` duplicate `--model` flag) that wasn't covered by the 3-test BATS suite — the suite tested dispatch routing but not arg passthrough edge cases
- All 7 Copilot findings were substantive (no noise): 1 code bug, 2 BATS isolation issues, 4 docs accuracy gaps
- BATS suite was already in place at PR time, making it easy to confirm the `--model` fix didn't break existing dispatch tests
- CHANGE.md merge conflict was caught and resolved cleanly before PR creation — no dirty-PR / CI-skip incident

## What Went Wrong

- PR created in the wrong repo first (k3d-manager PR #70, not lib-foundation) — closed immediately after user caught it; no wasted CI
- `_ai_agent_review` shipped without `--model` dedup logic; the BATS tests didn't exercise the explicit-`--model` path, so the bug passed review
- Docs said "All functions require `K3DM_ENABLE_AI=1`" — this was stale from before the gate removal and was never updated when `_copilot_review` lost its gate
- Subtree example in docs called `_copilot_review` directly (not `_ai_agent_review`) and didn't note the hardcoded scope limitation

## Process Rules Added

| Rule | Where |
|------|--------|
| lib-foundation PR must precede k3d-manager subtree pull PR — never open the consumer PR first | CLAUDE.md (reminder: two-repo subtree discipline) |
| When writing BATS for a dispatch function, include a test for the explicit-flag-wins path | implicit — add to spec template |

## Decisions Made

- `_copilot_auth_check` still had the `K3DM_ENABLE_AI` gate at v0.3.17 merge time — subsequently identified as a bug and fixed in v0.3.18 (gate removed; auth now checked via env tokens → `apps.json` → `gh auth status`)
- `_copilot_scope_prompt` hardcodes "k3d-manager repository" — documented as a known limitation; consumers must override the function if using lib-foundation in a different project context
- `_ai_agent_review` explicit `--model` takes precedence over `AI_REVIEW_MODEL` env default — forward-looking design for callers that want per-invocation model selection without changing global env

## Theme

v0.3.17 established the generic AI review abstraction layer in lib-foundation: `_ai_agent_review` as the stable public interface and `_copilot_review` as a Copilot-specific backend. The milestone also cleaned up a category bug — a lib-foundation backend was checking a consumer-specific env var (`K3DM_ENABLE_AI`), violating the library/consumer boundary. Copilot's review pass added meaningful value: the `--model` dedup fix and BATS isolation improvements were genuine correctness issues that the original author missed. The two-repo PR ordering mistake (k3d-manager PR before lib-foundation) was caught by the user and corrected quickly, reinforcing the subtree-discipline rule.
